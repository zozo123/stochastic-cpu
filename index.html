The Inference Substrate.

A field report on the cloud that's being built underneath Software 3.0. From the agent-hour to the prompt-keyed cache to the three signatures on every 2030 merge.

by Yossi Eliaz · Principal Engineer at Incredibuild & Founder, islo.dev · Tel Aviv · 2026

#§ 01 · PreludeThe software we used to write.

The way we make software changed three times in a decade. We've only named two of them.

In 2017, Andrej Karpathy called the first shift Software 2.01. The code stopped being typed by humans. It became weights, learned from data, compiled by gradient descent. We argued about whether to call it programming. The argument lost. The weights kept shipping.

In 2025, he gave the rest of it a name. Software 3.02. Your prompts are programs. English is the language. The compiler is a GPU cluster, and the runtime is an LLM. The Iron Man suit and the autonomy slider were the same sentence.

That's two paradigms changing in eight years. Production. Consumption. Both renamed.

The third change is the one nobody named yet. The cloud underneath. The substrate.

That's what this essay is about.

#§ 02 · Software 3.0, plainlyThree sentences.

Karpathy said it plainly. Your prompts are programs.

That's one. Here are the other two.

The LLM is a computer. Not a model. A computer. The same way a CPU is a computer — a substrate that runs instructions and returns results. Except the instructions are English, and the results are sometimes wrong, and the memory is the context window.

The autonomy slider exists. You can keep the human in the loop, or you can hand the keys over. There's no right answer. There's an axis.

That's it. Three sentences.

Everyone else has been saying the same thing in worse words. Agentic. Copilot. Workflow automation. Vibe coding4. They're all reaching for one of Karpathy's three sentences and missing.

The reason this matters here — in a cloud essay, not an AI essay — is that all three sentences land on the infrastructure. Prompts are programs, which means somebody has to host them. The LLM is a computer, which means somebody has to bill for its cycles. The autonomy slider exists, which means the cloud has to know whether the workload is a human typing or an agent running for nine hours.

None of which the current cloud knows. None of which the current cloud was asked.

#§ 03 · What brokeWhat containers got wrong about agents.

Containers were built for humans. Agents broke them in eighteen months.

Containers assume a request. Agents run a loop. That's the whole problem.

A SWE-bench Verified trajectory6 is not a request. It is a thirty-minute conversation with two hundred tool calls — read a file, run a test, edit a file, run a test, install a dep, recompile, edit again. Two hundred small mutations across one long arc. Kill the container halfway through and the work dies with it. Restart cold and the model spends ten thousand tokens re-learning where it left off.

Stateless 12-factor was the right answer for a request/response web. It is the wrong answer for an agent.

What breaks, in order:

Cold start. A 400ms cold start is invisible when a human waits. It is unbearable when an agent fires two hundred sandboxes in a row to test two hundred edits. The aggregate cold-start tax becomes the biggest single line item on the bill.

State. A container is a fresh slate every time. An agent is a slate with thirty minutes of marks on it. Throw the marks away between turns and you've thrown away the work.

Pricing. Containers are billed per second of compute. Agents are billed — should be billed — per agent-hour, snapshot-GB-month, eval token. The shapes don't overlap.

CI. CI queues built for ten PRs a day collapse at a thousand. Cursor's background agents14, Shopify's River9, Devin's autonomous pool — each of them is now opening hundreds of PRs a week. The bottleneck is no longer the human reviewer. It's the build farm.

Cache. Bazel hashes the bytes. Agents change the bytes without changing the meaning. The cache misses every time.

The container era isn't ending because containers are bad. It's ending because the workload changed and the workload doesn't fit anymore.

#§ 04 · The new unitThe agent-hour.

The agent-hour is the new CPU-hour. Bill accordingly.

Anthropic split the bill in June. That was the leading indicator7.

One SKU for human-seat usage — chat, copilot, the thing a person sits in front of. Another SKU for headless agent usage — the API pool that runs unattended, mutates files, opens PRs, sleeps until the next webhook. Different rates. Different rate limits. Different telemetry.

Once one vendor splits the bill, the rest follow. Modal already prices its sandboxes at roughly three times the rate of its general serverless tier — same hardware, different SKU, different expectation. E2B sells snapshot-resume as a tier above raw container minutes. Replit's Agent runtime quotes minutes, not requests5. Cursor's background agent has its own line item14.

They're all reaching for the same SKU under different names. The shape is: a long-running, stateful, autonomous workload, billed by wall-clock time, with surcharges for snapshot storage and accelerator access. Call it the agent-hour.

By 2027 the agent-hour is a published line item at all three hyperscalers. Not a private SKU. Not a special-deal contract term. A page on the pricing site, next to m7i.large.

The follow-on effect is what changes the architecture. If the agent-hour is a billable unit, then everything else in the stack — scheduling, autoscaling, snapshotting, eviction — gets rebuilt around it. Kubernetes wasn't built for nine-hour pods that mutate their disk. Something else gets built. Something else is being built right now.

And the FinOps team finally has the right number to optimize against. Not requests per second. Agent-hours per merged PR. That's the cost ratio that matters.

#§ 05 · The opinionated flagThe two-level cache.

Content-addressed caches lose to prompts. Add a second key.

Content-addressed caches were built for humans typing. Humans typing are no longer the bottleneck.

Bazel, Buck, Nix, sccache — all of them hash the input bytes and look up the output. A heroic engineering achievement. The reason your CI doesn't take six hours. The reason the monorepo works.

Here's what changes. An LLM, asked to fix a bug, will produce a hunk of code that is semantically equivalent to a previous hunk but byte-different. A trailing newline. A renamed local variable. A whitespace-collapsed import. The content hash flips. The cache misses. The whole subtree recompiles.

You can fix it by canonicalizing — sort imports, normalize whitespace, run a formatter before hashing. That helps. It doesn't solve it. Two prompts asking the model to "refactor for clarity" will produce different code with the same intent. There's no formatter that catches that.

The fix is two layers.

Bottom layer: content-addressed, exactly as today. Hash the bytes, look up the artifact. Keep Bazel. Keep Nix. Don't rebuild what works.

Top layer: prompt-prefix-keyed. The cache key is the model's input prefix — the system prompt, the tool definitions, the file contents that didn't change. Anthropic's prompt cache already does this at the inference layer7. Ninety percent off the cached prefix. The trick is to extend the same idea up into the build cache, the test-runner cache, the CI artifact cache.

Two keys on every cache lookup. Content at the bottom. Prompt at the top.

No product ships this end-to-end yet. The first one that does becomes the BuildKit of the agent era. We're building toward it at Incredibuild12. We're not the only ones.

This is the part of the essay that's a flag, not a forecast. By the time you're reading this, somebody has the cache. By 2027, three vendors have the cache. By 2030, the cache is part of the substrate the way Docker layers are part of the substrate today — invisible, assumed, infrastructure.

#§ 06 · A number, carefullyThe Tokio benchmark.

At Incredibuild we measured forty-six seconds. Cached, twelve. Hot, three.

I'll show you one benchmark. We ran it at Incredibuild12.

The workload is cargo test --workspace on Tokio — a Rust async runtime with a well-known dependency graph. Eight hundred crates. Three minutes of compile work on a clean machine. The kind of project where caching matters because the project is large enough to feel it.

Cold (no cache)46.0 s
Cached (warm, deps stable)12.0 s
Hot (cache + warm runner)3.0 s
Incredibuild internal measurement, 2025. Repro: github.com/Incredibuild-RND/buildrunner-on-islo

The interesting number is not 46. It's 3.

Three seconds is the number an agent gets when it edits one file in a large codebase and reruns the suite. Nothing else moved. The cache has the dependency graph, the test binaries, the build artifacts. Forty-seven of the fifty test crates are untouched. They don't recompile. They don't relink. They don't rerun.

This is the cache hit rate you get when humans edit code at a normal cadence — a commit every fifteen minutes, a PR every two hours, a merge every day. The numbers don't change much under that load.

The numbers change when the cadence changes. An autonomous agent making fifty file edits in an hour, running the tests after each one, looks nothing like a human committer. It looks like a stress test of the cache. Forty-seven hits and three misses per edit. That's the cache shape that decides whether the agent-hour costs four dollars or forty.

The bill in 2030 is not won at the rate card. It is won at the cache hit rate.

#§ 07 · Who signed the codeThree signatures on every merge.

By 2030, three signatures on every merge. The human. The agent. The platform.

By 2030 you won't merge without three signatures. Not in finance. Not in healthcare. Not anywhere it matters.

Sundar said three quarters of Google's code was already AI-generated by late 202511. The number sounded uncomfortable. It also sounded like an under-count. Once you include the small refactors and the test boilerplate and the migration patches and the README updates, three quarters is conservative.

The compliance question stops being "did this build reproduce?". It becomes "which model wrote which hunk, and which prompt produced it?".

The three signatures are:

The human. Same as today. The reviewer who clicked approve. A real person with a real GPG key. Optional in many shops. Required in regulated ones.

The agent. Model name, model version, prompt hash, trajectory ID. Not the agent's personality. Its identity. Cosign + Sigstore15 already has the primitives. SLSA v1.1 extends them to AI-generated artifacts. The standards are written. The deployment is partial.

The platform. The cloud signs the runtime — which container image, which CPU vendor, which TEE, which key in which HSM. AWS Nitro, Azure CC, GCP Confidential Space already sign builds. By 2028 the signature extends to the agent runtime: "this code was emitted by Claude 4.7 running in an islo.dev microVM with attested hardware and prompt-cache hit ratio 0.92." That's the provenance line item.

This is the part of the essay that isn't speculative. It's already happening in the shops that ship to FedRAMP and the EU CRA. By 2028 it is table stakes. By 2030 it is invisible — every merge has it, nobody talks about it, the same way TLS is in every URL bar.

The corollary: code without provenance becomes the new code without tests. Allowed. Tolerated. Quietly distrusted.

#§ 08 · The shapeSix years. Six milestones.

Here's the shape, year by year. The leading indicators are dated. The forecasts are reasoned. None of this is fiction — every milestone has a real public artifact to point at, or it isn't on the list.

  1. 2025
    The naming.
    Karpathy gives the YC keynote on June 17. "Your prompts are now programs that program the LLM."2 Devin 2.0 reports 45.8% on SWE-bench Verified6. Cursor and Replit Agent become the default tool surfaces for AI-assisted work.
  2. 2026
    The billing split.
    Anthropic separates human-seat usage from headless agent usage on June 15. Modal prices sandboxes at 3× the serverless rate5. Shopify's "Under the River" agent merges 12.5% of all PRs in one month9.
  3. 2027
    The agent-hour ships.
    AWS publishes a Bedrock + Fargate agent-hour SKU. More than half of merged PRs at the top-ten AI-native companies are agent-authored. Prompt-keyed build caches ship as a category for the first time.
  4. 2028
    The provenance year.
    The EU CRA forces signed agent attestations on regulated deploys. Sigstore + SLSA v1.1 become table stakes15. Every merge in finance and healthcare names the model that wrote it.
  5. 2029
    The fleet replaces the container.
    Persistent agent fleets — three to ten agents per active repo, each in a hot microVM — become the default deployment shape. Cursor-style background agents14 generalize across every major vendor.
  6. 2030
    The bill flips.
    Legacy stateless compute drops to under 10% of the AI-native cloud bill. Agent-hours and snapshot storage dominate. Kubernetes survives as an agent-fleet orchestrator, not a request-router.

#§ 09 · The destinationThe cloud bill, 2030 edition.

The container era is not over. It just got demoted.

The cloud bill in 2030 doesn't look like the cloud bill in 2024. Not even close.

The top of the invoice — the line that was once EC2 / Compute at 60% of the total — drops to a fifth of what it was. Stateless compute didn't disappear. It got demoted. It runs the request paths. It runs the gateways. It does not run the agents.

Here's the shape, by share of total bill, for an AI-native company in 2030:

Inference tokens (model API)40 %
Agent-hours (persistent fleets)25 %
Snapshot & cache storage (GB-mo)15 %
Eval compute (offline + online)10 %
Egress & networking5 %
Legacy stateless compute5 %

Two things to notice.

First, the bill is no longer dominated by what the cloud does for you. It is dominated by what the agents cost to keep running. Inference is the labor cost. Agent-hours are the rent. Cache and snapshot are the warehouse. The proportions look more like a manufacturing P&L than a SaaS one.

Second, eval compute is a published line item. By 2030, eval-time spending is on the same page as serve-time spending. You don't get to ship the model without the evals running every hour, and the evals run on your bill.

Kubernetes is still there. It survived. It just stopped being the load-bearing wall. It became the agent-fleet scheduler — the layer that decides which microVM holds which agent's snapshot, which evals run when, which background tasks defer until off-peak. That's a real job. It's a smaller job than running the whole internet.

The container era isn't over. It just got demoted.

#§ 10 · One more thingThe substrate.

Software 1.0 was written by humans for machines.

Software 2.0 was written by data for weights.

Software 3.0 is written by humans for models that write for machines.

The thing underneath has to change. The substrate has to change.

It already is.

The agent-hour is being named. The bill is being split. The cache is being rebuilt with two keys. The microVM is replacing the container for the workloads that mutate. The signatures are being attached. The eval line item is appearing on the invoice. Each of these is happening in a different shop, on a different timeline, with a different vendor pitching a different SKU.

None of them are the whole answer. All of them are the same answer, told from different sides of the same room.

If you're building anything where an agent is the producer and the cloud is the substrate — and that's an expanding set, not a shrinking one — the next eighteen months decide whether you're early or late.

The substrate is shifting. It's already shifted. You're standing on it.
